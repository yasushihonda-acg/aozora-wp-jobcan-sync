## 3. フィードバック収集

### 推奨: Google フォーム (固定項目)
理由: 自由記述だけだと「なんとなく違う」が来て手戻り増。項目を固定して定量+定性で集める。

フォーム項目例:
1. **トンマナ一致度** (5 段階)
2. **情報の過不足** (自由記述)
3. **応募導線の分かりやすさ** (5 段階 + コメント)
4. **モバイル表示の見やすさ** (5 段階)
5. **優先度高い修正点** (自由記述、最大 3 つ)
6. **本リビジョンを「方向性 OK」とするか** (Yes/No/もう一度議論したい)

### 代替: 議事録 / メール
オンラインミーティングで画面共有しながら直接確認も可。その場合は議事録を Phase A 担当者がまとめて Git にコミット。

## 4. リビジョン管理

- mockup/ は git 管理 (Phase A 公開段階で git init)
- リビジョンタグ: `phase-a-v0.1`, `phase-a-v0.2`, ... を git tag で付与
- 変更点サマリーを各リビジョンで README に追記:
  ```markdown
  ### phase-a-v0.2 (2026-06-XX)
  - 求人カードの給与表示を月給/時給で区別
  - CTA ボタン色を #00c4cc → #2a6a78 にトーン変更 (決裁者意見反映)
  - スタッフインタビューを 2 名→ 3 名に
  ```

## 5. ゲート 1 (決裁)

### チェック
- `docs/specs/acceptance-criteria.md` の全項目消化
- 各項目に決裁者署名 (口頭 → 議事録 / メール / ドキュメントコメント)

### 署名手段
- **単独決裁者の場合**: メール返信「ゲート 1 承認」+ 議事録への記録
- **複数決裁者の場合**: 全員のメール返信受領を必須条件 (全員一致)
- **意見が割れた場合**: 拡大会議 → 主担当者の最終判断 + 議事録

### 拒否された場合
- リビジョンに戻る (Week 2-3 想定の枠内なら継続)
- 想定リビジョン回数 (3 回) を超えた場合は Phase A スコープ自体を再検討
