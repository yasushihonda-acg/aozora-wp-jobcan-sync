# Acceptance Criteria — Phase A 決裁ゲート 1

> Codex セカンドオピニオン (2026-06-16) でデザインだけでなく求人項目・SEO・応募導線・障害時対応・運用体制まで承認範囲を拡張済。

---

## ゲート 1 を通過する条件

下記 A〜F の全項目で「決裁者の合意」が得られた場合のみ、Phase B (WordPress 本番構築) に進む。

### A. デザイン

- [ ] **A-1**: ACG コーポレートサイト (`https://aozora-cg.com/`) とのトンマナ一致 (色・フォント・余白の 5 段階評価で 4 以上)
- [ ] **A-2**: レスポンシブ表示 (iPhone SE / iPad / PC) で崩れなし、Chrome / Safari / Edge 互換
- [ ] **A-3**: デザイントークン表 (`docs/specs/design-tokens.md` v0.2) が確定済 (アクセント色の最終決定を含む)

### B. 情報設計・項目

- [ ] **B-1**: 募集職種カテゴリーが現状ジョブカン公開求人 (16 カテゴリー) と過不足なく対応
- [ ] **B-2**: 求人一覧カードの表示項目 (職種 / 勤務地 / 給与 / 雇用形態 / 募集要約) が決裁者合意
- [ ] **B-3**: 求人詳細の項目セット (仕事内容 / 必須資格 / 歓迎条件 / 給与 / 勤務時間 / 休日 / 福利厚生 / 応募方法) が決裁者合意
- [ ] **B-4**: フィルタ UI (職種 × 勤務地 × 雇用形態) の組み合わせ仕様が確定

### C. 応募導線

- [ ] **C-1**: 応募ボタン → ジョブカン応募フォーム URL パターン (`/aozora/entry/new/{job_id}`) に正しく繋がる
- [ ] **C-2**: 応募フォーム障害時の **代替導線 (電話 / 問い合わせ)** が明示されている
- [ ] **C-3**: **CTA 固定表示** (モバイルでもスクロール時に応募ボタンが見える)
- [ ] **C-4**: 応募完了率を測定する仕組み (GA4 event タグ / クリック計測) の方針が確定

### D. SEO 方針

- [ ] **D-1**: **canonical 方針** 確定: `recruit.aozora-cg.com` を canonical とするか、ジョブカン元ページとするか
- [ ] **D-2**: **JobPosting 構造化データ** (schema.org) 実装方針が合意 — 求人詳細に JSON-LD 出力
- [ ] **D-3**: ジョブカン元ページとの **重複対策** (旧ページの noindex 化を依頼するか、放置するか)
- [ ] **D-4**: サイトマップ (`sitemap.xml`) と Search Console 登録の予定
- [ ] **D-5**: メタ情報 (title / description / OGP / Twitter Card) の方針

### E. 運用・障害対応

- [ ] **E-1**: **同期方式の最終決定** (要 ジョブカン公式照会結果 — `docs/specs/sync-strategy.md`)
- [ ] **E-2**: 同期障害時の手動運用フォールバック手順 (CSV 手動更新等) が手順書化
- [ ] **E-3**: 応募リンク死活監視の方針 (link checker 日次)
- [ ] **E-4**: 求人取得から WP 反映までの計測可否 (応募完了率を測れるか) の方針確定
- [ ] **E-5**: 運用体制 (誰が日々の運用・障害対応を担うか、エスカレーション先)

### F. 法務・組織

- [ ] **F-1**: **個人情報棚卸し** 完了: WP に保存される情報 / アクセスログ / GA4 / 応募リンククリックログを分類
- [ ] **F-2**: 上記分類結果に基づき **ismap 適用範囲を確定** (求人サイト本体に適用するか否か)
- [ ] **F-3**: プライバシーポリシー / 利用規約 / 応募者向け個人情報取扱い同意の方針
- [ ] **F-4**: WP ホスティング選定確定 (`docs/specs/hosting-comparison.md` の比較を元に組織判断)

### G. 決裁

- [ ] **G-1**: 上記 A〜F について **決裁者の口頭またはメール承認** 取得
- [ ] **G-2**: 議事録に決裁内容を記録、責任分界 (修正コミット担当・最終リリース判断者) を明記

## チェック方法

| 項目 | 確認手段 |
|---|---|
| A デザイン | ローカル Chrome/Safari/Edge + DevTools レスポンシブ + aozora-cg.com サイドバイサイド |
| B 情報設計 | モック画面 + recruit.jobcan.jp/aozora 実データとの突合表 |
| C 応募導線 | 応募ボタンクリック → URL パターン確認、CTA 固定表示は iPhone SE で目視 |
| D SEO | モック HTML の `<head>` / 構造化データ JSON-LD / canonical タグを目視 |
| E 運用 | 別途運用手順書のドラフト (`docs/specs/runbook-draft.md` — Phase A 内で起こす) |
| F 法務 | 個人情報棚卸しシート + 法務レビュー結果 |
| G 決裁 | メール / 議事録 |

## NG パターン (再リビジョン → 再ゲート)

以下の場合は Phase B に進まず Phase A を継続:
- 決裁者からの「もう一度方向性を変えてほしい」コメント → リビジョン (想定 2〜3 回内)
- ジョブカン公式照会の回答が「不可」または「曖昧」のまま → 同期方式 (E-1) 確定不可、代替案検討
- 個人情報棚卸し結果でホスティング判断が割れた → 法務含む拡大会議で確定
- SEO 方針で重複対策が組織内で合意できない → SEO 専門家を交えて再判定
